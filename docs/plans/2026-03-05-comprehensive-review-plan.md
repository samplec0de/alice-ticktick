# Комплексное ревью Alice-TickTick: План реализации

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Найти и исправить баги, UX-проблемы и code smell-ы через коллегиальный многоагентный процесс с critic-агентом.

**Architecture:** Team-lead координирует 3 reviewer-ов (UX, Logic, Code) + 1 critic. Reviewer-ы анализируют параллельно → critic атакует каждый отчёт → team-lead проводит дискуссию по спорным issues → implementer создаёт PR на каждый принятый issue.

**Tech Stack:** Python 3.12, aliceio, httpx, rapidfuzz, pydantic, ruff, mypy, pytest, uv

---

## Фазы плана

1. **Фаза 1:** Параллельный анализ (3 reviewer-а одновременно)
2. **Фаза 2:** Critic атакует каждый отчёт
3. **Фаза 3:** Дискуссия по спорным issues, финальный список
4. **Фаза 4:** Реализация фиксов (по одному PR на issue/группу)

---

## Фаза 1: Параллельный анализ

### Task 1: UX Reviewer — анализ ответов и диалогов

**Агент:** `ux-reviewer` (general-purpose)

**Файлы для чтения:**
- `alice_ticktick/dialogs/responses.py` — все тексты ответов
- `alice_ticktick/dialogs/handlers.py` — как формируются ответы
- `alice_ticktick/dialogs/router.py` — welcome/goodbye

**Что анализировать:**

1. **Стиль и последовательность** — Алиса говорит "Готово!", "Добавила!", "Создала" — непоследовательный стиль (мешать от 1-го и 3-го лица)
2. **TTS читаемость** — знаки препинания, кавычки `«»` vs `""`, скобки — как это звучит вслух
3. **Лимит 1024 символа** — найти handlers где списки задач не обрезаются и могут превысить лимит
4. **Ошибочные сценарии** — что говорит Алиса при `API_ERROR`, `TASK_NOT_FOUND`, пустом слоте — достаточно ли информативно
5. **Брифинги** — утренний/вечерний: формулировки, информативность, что если одновременно много задач И просроченные
6. **HELP текст** — актуален ли, все ли команды перечислены
7. **pluralize_tasks** — корректна ли логика для всех чисел

**Формат отчёта:**

```
## UX Report

### Issue UX-1: [название]
**Серьёзность:** critical / high / medium / low
**Файл:** responses.py:35
**Проблема:** [описание]
**Пример:** [конкретный текст]
**Предлагаемый фикс:** [новый текст или подход]

### Issue UX-2: ...
```

### Task 2: Logic Reviewer — анализ интентов и edge cases

**Агент:** `logic-reviewer` (general-purpose)

**Файлы для чтения:**
- `alice_ticktick/dialogs/router.py` — порядок интентов, disambiguation
- `alice_ticktick/dialogs/handlers.py` — обработка слотов, FSM
- `alice_ticktick/dialogs/intents.py` — извлечение слотов
- `alice_ticktick/dialogs/nlp/date_parser.py` — парсинг дат
- `alice_ticktick/dialogs/states.py` — FSM состояния
- `tests/test_handlers.py`, `tests/test_handlers_filtering.py` — что уже покрыто

**Что анализировать:**

1. **Disambiguation add_subtask vs create_task** — условие `_SUBTASK_KEYWORDS` — есть ли фразы, которые не попадают
2. **`_try_parse_checklist_command`** — regex покрывает ли все паттерны? "добавь элемент X к чеклисту задачи Y"?
3. **FSM delete — cold start** — если Lambda перезапустилась между "удали" и "да", что происходит?
4. **`_MAX_CONFIRM_RETRIES = 3`** — что происходит после 3 неверных ответов? Состояние очищается?
5. **list_tasks без даты** — что если пользователь скажет "покажи задачи" без даты — что вернёт интент?
6. **overdue + list_tasks пересечение** — просроченные задачи попадают ли в list_tasks на "сегодня"?
7. **Пустые слоты** — что если task_name = "" или состоит из стоп-слов (STOPWORDS в handlers)?
8. **Timezone** — откуда берётся timezone пользователя? Что если не передаётся?
9. **complete_task fuzzy** — если нашли задачу с низким score — отмечаем без подтверждения?

**Формат отчёта:**

```
## Logic Report

### Issue L-1: [название]
**Серьёзность:** critical / high / medium / low
**Файл:** router.py:137
**Проблема:** [описание]
**Сценарий воспроизведения:** [шаги]
**Предлагаемый фикс:** [подход]

### Issue L-2: ...
```

### Task 3: Code Reviewer — анализ структуры и качества кода

**Агент:** `code-reviewer` (general-purpose)

**Файлы для чтения:**
- `alice_ticktick/dialogs/handlers.py` (весь файл, 2212 строк)
- `alice_ticktick/dialogs/intents.py`
- `alice_ticktick/dialogs/nlp/date_parser.py`
- `alice_ticktick/ticktick/client.py`

**Что анализировать:**

1. **handlers.py размер** — 2212 строк в одном файле: какие группы функций можно выделить в отдельные модули без нарушения логики
2. **Дублирование фильтров** — `_apply_task_filters` vs аналогичная логика в `handle_overdue_tasks`, `handle_project_tasks` — есть ли дублирование?
3. **`except Exception`** — где ловятся исключения без достаточного логирования или с проглатыванием ошибок
4. **`Any` в сигнатурах** — `intent_data: dict[str, Any]` — можно ли типизировать строже
5. **Мёртвый код** — функции/переменные/импорты которые не используются
6. **Магические числа/строки** — где есть литералы без констант (кроме уже вынесенных)
7. **asyncio.gather** — используется ли параллельность там, где можно; где последовательные await можно распараллелить
8. **Тесты** — есть ли handlers/логика без тестового покрытия (сравни с test_*.py файлами)

**Формат отчёта:**

```
## Code Report

### Issue C-1: [название]
**Серьёзность:** critical / high / medium / low
**Файл:** handlers.py:450-623
**Проблема:** [описание]
**Предлагаемый фикс:** [подход с примером кода]

### Issue C-2: ...
```

## Фаза 2: Critic атакует отчёты

### Task 4: Critic атакует UX Report

**Агент:** `critic` (general-purpose)

**Получает:** UX Report от `ux-reviewer`

**Задача:** Для каждого issue в UX Report найти слабое место. Формат атаки:

```
## Critique of UX Report

### UX-1: [принято / оспорено / уточнение]
**Аргумент:** [почему issue не так критичен, или reviewer упустил что-то, или предложенный фикс неверен]

### UX-2: ...

### Пропущенные issues (найдено critic-ом):
### Issue UX-N: [что reviewer не заметил]
```

Critic должен быть строгим: если issue очевидный и верный — написать "принято, нечего оспорить". Не критиковать ради критики.

### Task 5: Critic атакует Logic Report

**Агент:** `critic` (тот же агент, после Task 4)

**Аналогично Task 4**, но для Logic Report. Особое внимание:
- Является ли описанный сценарий воспроизводимым?
- Предложенный фикс не ломает другие интенты?
- Достаточно ли предложено тестов?

---

### Task 6: Critic атакует Code Report

**Агент:** `critic` (тот же агент, после Task 5)

**Аналогично**, но для Code Report. Особое внимание:
- Является ли рефакторинг YAGNI или реально нужен?
- Нет ли оверинжиниринга в предложенных фиксах?
- Есть ли риск регрессии при предложенных изменениях?

## Фаза 3: Дискуссия и финальный список

### Task 7: Team-lead сводит финальный список issues

**Агент:** team-lead (я)

**Шаги:**

1. Собрать все issues из трёх отчётов
2. По каждому спорному (оспоренному critic-ом) — запросить ответ от соответствующего reviewer
3. Принять решение: **включить / отклонить / включить с изменённым фиксом**
4. Присвоить финальный приоритет: P0 (критично, блокирует) → P1 (важно) → P2 (улучшение) → P3 (nitpick)
5. Сгруппировать по PR-ам (один PR = один логический блок изменений)

**Сохранить в:** `docs/plans/2026-03-05-review-issues.md`

## Фаза 4: Реализация фиксов

### Task 8+: По одному PR на каждую группу P0/P1 issues

**Для каждого принятого issue:**

**Шаг 1: Написать failing test**
```bash
uv run pytest tests/test_handlers.py::test_<issue_name> -v
# Expected: FAIL
```

**Шаг 2: Реализовать фикс**

Минимальное изменение, не трогать несвязанный код.

**Шаг 3: Запустить полный CI**
```bash
uv run ruff check .
uv run ruff format --check .
uv run mypy alice_ticktick/
uv run pytest -v
```

**Шаг 4: Коммит и PR**
```bash
git add <files>
git commit -m "fix: <описание>"
gh pr create --title "<title>" --body "..."
```

**P2/P3 issues** — создать отдельный PR с пометкой "nice to have", не блокировать.
